[Geek笔记]复杂度分析（三）：四个神奇的复杂度

这篇文章来谈一谈四个复杂度分析方面的点：最好情况时间复杂度、最坏情况时间复杂度、平均情况时间复杂度、均摊时间复杂度。

最好、最坏情况时间复杂度

首先还是来看一段代码：

// n 表示数组 array 的长度
int find(int[] array, int n, int x) {
int i = 0;
int pos = -1;
for (; i < n; ++i) {
if (array[i] == x) pos = i;
}
return pos;
}

这段代码实现的功能是，在一个数组中寻找一个变量x，并返回它的位置。很明显，这里的复杂度为O(n)。但事实上，我们并不一定需要遍历完整个数组。如果我们在前面的位置发现了这个元素，就可以直接返回了：

// n 表示数组 array 的长度
int find(int[] array, int n, int x) {
int i = 0;
int pos = -1;
for (; i < n; ++i) {
if (array[i] == x) {
pos = i;
break;
}
}
return pos;
}

就像这样。如果要找的元素在第一位，则只需要循环一次就可以找到它并结束循环。而若数组中没有这个元素，就仍需要循环n次。这两种情况下的时间复杂度分别为O(1)和O(n)。 这两种情况就分别对应着最好情况时间复杂度和最坏情况时间复杂度。最好时间复杂度就是在最理想的情况下，执行这段代码的时间复杂度。同理，最坏时间复杂度就是在最糟糕的情况下，执行这段代码的时间复杂度。

平均情况时间复杂度

事实上，最好和最坏的情况都不是那么经常发生的。那我们有什么更好的方式来表示时间复杂度吗？ 在上面的第二个例子中，我们假设元素在与不在数组的概率都为1/2。那么，在数组中任一位置的概率就为1/2n。因此，需要循环的次数的平均值就为： 1*1/2n+…..+n*1/2n+n*1/2=(3n+1)/4 这个其实就是一个加权平均数，算出来的时间复杂度就是平均情况时间复杂度。我们可以看到，这里算出来的时间复杂度仍然为O(n)。

均摊时间复杂度

还是先看一段代码：

// array 表示一个长度为 n 的数组
// 代码中的 array.length 就等于 n
int[] array = new int[n];
int count = 0;

void insert(int val) {
if (count == array.length) {
int sum = 0;
for (int i = 0; i < array.length; ++i) {
sum = sum + array[i];
}
array[0] = sum;
count = 1;
}

array\[count\] = val;
++count;

}

这段代码的意思是，如果数组没有满，就直接将数据插入数组；如果满了，就先将数组遍历求和，将和存入数组第一个元素，再插入后面的元素。很明显，最好和最坏时间复杂度分别为O(1)和O(n)，而平均复杂度为O(1)。像这种大部分情况复杂度很低，极端情况下复杂度很高的代码，我们可以将复杂度高的部分分摊到低的部分。因此，一般均摊时间复杂度等于最好情况时间复杂度。 （反正也不会用，溜了溜了）

[Geek笔记]复杂度分析（二）：一些栗子

上一篇文章中讲到，通过大O来表示时间复杂度。常见的复杂度有以下几种： 常量阶O(1)、对数阶O(logn)、线性阶O(n)、线性对数阶O(nlogn)、k次方阶O(n^k)、指数阶O(2ⁿ)、阶乘阶O(n!)。 我们可以将上述量级分为两类：多项式量级和非多项式量级。其中只有指数阶和阶乘阶为非多项式量级。当数据规模越来越大时，非多项式量级算法执行时间将急剧增长，这种算法是非常低效的，一般不使用。下面看看几种常见的多项式时间复杂度。 1.O(1) 这里的”1”指的不是只有一行代码，而是理解成随着n的增大，每个语句所只执行1次。例如：

int i = 8;
int j = 6;
int sum = i + j;

这里有三行代码，但它的时间复杂度仍然是O(1)，不是O(3)。也就是说，无论n是1000、10000、甚至是更大，它的时间复杂度仍然是O(1)而不是O(n)。一般来说，只要算法中不存在循环、递归，其时间复杂度就是O(1)。 2.O(logn)、O(nlogn) 还是举个栗子：

i=1;
while (i <= n) {
i = i * 2;
}

很容易发现，第三行代码被执行的次数最多，只要知道这段代码被执行了多少次，就能知道时间复杂度了。事实上，这段代码的意思就是求出k个2相乘，然后当结果大于n的时候结束循环。所以，k=log₂n的。而对数之间是可以相互转换的，我们就忽略掉对数的底，全部写成O(logn)。 那么根据乘法原则，把上面这段代码循环n次，时间复杂度就变成了O(nlogn)。归并排序、快速排序的复杂度都是这个。 3.O(m+n)、O(m*n) 看一段和之前不太一样的代码：

int cal(int m, int n) {
int sum_1 = 0;
int i = 1;
for (; i < m; ++i) {
sum_1 = sum_1 + i;
}

int sum_2 = 0;
int j = 1;
for (; j < n; ++j) {
    sum\_2 = sum\_2 + j;
}

return sum\_1 + sum\_2;

}

这里的m、n是两个不同的数据规模，而且我们并不知道谁的规模大。这时候就不能用加法原理来省去其中一个了。事实上，这段代码的时间复杂度为O(m+n)。同样，若是循环嵌套等，时间复杂度为O(m*n)。 看过这么多时间复杂度的例子，接下来简单提一下空间复杂度。仍然是用一个例子：

void print(int n) {
int i = 0;
int[] a = new int[n];
for (i; i <n; ++i) {
a[i] = i * i;
}

for (i = n-1; i >= 0; --i) {
    print out a\[i\]
}

}

第2行中，申请了一个空间存储变量i，而第三行中，申请了n个空间来存一个数组。所以这段代码的空间复杂度为O(n)。常见的空间复杂度就是O(1)、O(n)、O(n²)。

[Geek笔记]复杂度分析（一）：入了个门

开始更数据结构与算法部分的学习笔记了。开头带[Geek笔记]的都是“极客时间”专栏的学习笔记，其它的则是从教材、习题、acm等而来，基本上都会有这种前缀。 事实上这个更新本来会来的更慢，然而我再一次“被迫”去计算某算法的复杂度。那就只好提前开始了（其实早就该开始了…) 首先从一段最简单的代码开始：

int cal(int n) {
int sum = 0;
int i = 1;
for (; i <= n; ++i) {
sum = sum + i;
}
return sum;
}

虽然说，cpu执行不同的语句所需要的时间不同，但因为是粗略估计，我们假定处理单行代码的时间相同，为1个单位时间。 在这段代码中，第2、3行代码需要一个单位的时间，而4、5行都运行了n次，也就是说这段代码的总运行时长为2n+2个单位时间，和n成正比，即T(n)=2n+2。 按照这个思路，我们可以再看一段代码：

int cal(int n) {
int sum = 0;
int i = 1;
int j = 1;
for (; i <= n; ++i) {
j = 1;
for (; j <= n; ++j) {
sum = sum + i * j;
}
}
}

这东西看起来又复杂了一点。我们还是假设执行每个语句的时间为1单位时间。 第2、3、4行代码中，每行都需要1个单位时间。第5、6行执行了n遍，总共需要2n个单位时间。第7、8行代码，由于是嵌套在循环当中的循环，执行了n²次，总共需要2n²个单位时间。也就是说，这段代码总共需要2n²+2n+3个单位时间，即T(n)=2n²+2n+3。 接下来出场的是大家喜闻乐见的O(f(n))。用一个公式来表示： T(n)=O(f(n)) 第一个例子中，T(n)=O(2n+2)，第二个例子中，T(n)=O(2n²+2n+3)。大O时间复杂度并不表示代码实际的运行时间，而是表示一个时间随数据规模增长的变化趋势。所以，也叫作渐进时间复杂度（asymptotic time complexity），简称时间复杂度。当n很大（可以想象趋近正无穷）时，公式中的低阶项和系数将不起作用，因此，也将上述两式简写为T(n)=O(n)、T(n)=O(n²)。 正因如此，用大O计算时间复杂度时，我们只用关注被执行最多的一段代码，以及量级最大的代码块的复杂度就可以了，其它的可以忽略掉。而如果在函数中嵌套调用函数，则这段代码的复杂度为两个函数复杂度的乘积（其实和循环嵌套有一丢丢像）。 今晚先这样吧，明天更一些例子和更高级的分析。

培训心得(一)

虽说题目里有个“一”，但是其实并不知道之后还有没有“二”甚至“三”。毕竟是自己第一次带培训，也不知道以后还会不会留在云麓谷，反正趁着刚结束了这个学期最后一次培训，先总结了再说吧。 其实现在看来，这个学期的培训其实是极其混乱的。一共培训了六次吧，一次线上的，一次围坐在一起敲代码的，算起来总共也就四次有效培训。第一次培训用了半个小时讲基础知识，一个小时讲爬虫。说实话，这样做效果是真的不好，直到现在都没有把学期初的任务完成。后来在他们的坚持下，开始从头开始讲。看起来都听懂了，但是实际用起来的话感觉还是有困难的。毕竟只花了三节课的时间，CI框架讲的极其简单，估计离上手还会有很长一段时间吧。 我在这些培训当中得到了些什么呢？我不确定。我感觉我只是把上个学期就应该掌握了的东西掌握了，自己水平也没有怎么提高过。但是如果不培训的话，可能早就离开了吧？还是得到了一些东西的，至少锻炼了自己与人交流的能力（虽然都是熟人（笑。可能这就是为什么给他们布置了考核内容之后，我也给我自己留了一道题。大概就是想练练手吧。 希望他们和前端的同学们合作愉快。也希望他们赶紧成长起来。这样无论我退或不退，压力应该都不会这么大了吧。加油。
