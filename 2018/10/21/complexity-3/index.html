[Geek笔记]复杂度分析（三）：四个神奇的复杂度

这篇文章来谈一谈四个复杂度分析方面的点：最好情况时间复杂度、最坏情况时间复杂度、平均情况时间复杂度、均摊时间复杂度。

最好、最坏情况时间复杂度

首先还是来看一段代码：

// n 表示数组 array 的长度
int find(int[] array, int n, int x) {
int i = 0;
int pos = -1;
for (; i < n; ++i) {
if (array[i] == x) pos = i;
}
return pos;
}

这段代码实现的功能是，在一个数组中寻找一个变量x，并返回它的位置。很明显，这里的复杂度为O(n)。但事实上，我们并不一定需要遍历完整个数组。如果我们在前面的位置发现了这个元素，就可以直接返回了：

// n 表示数组 array 的长度
int find(int[] array, int n, int x) {
int i = 0;
int pos = -1;
for (; i < n; ++i) {
if (array[i] == x) {
pos = i;
break;
}
}
return pos;
}

就像这样。如果要找的元素在第一位，则只需要循环一次就可以找到它并结束循环。而若数组中没有这个元素，就仍需要循环n次。这两种情况下的时间复杂度分别为O(1)和O(n)。 这两种情况就分别对应着最好情况时间复杂度和最坏情况时间复杂度。最好时间复杂度就是在最理想的情况下，执行这段代码的时间复杂度。同理，最坏时间复杂度就是在最糟糕的情况下，执行这段代码的时间复杂度。

平均情况时间复杂度

事实上，最好和最坏的情况都不是那么经常发生的。那我们有什么更好的方式来表示时间复杂度吗？ 在上面的第二个例子中，我们假设元素在与不在数组的概率都为1/2。那么，在数组中任一位置的概率就为1/2n。因此，需要循环的次数的平均值就为： 1*1/2n+…..+n*1/2n+n*1/2=(3n+1)/4 这个其实就是一个加权平均数，算出来的时间复杂度就是平均情况时间复杂度。我们可以看到，这里算出来的时间复杂度仍然为O(n)。

均摊时间复杂度

还是先看一段代码：

// array 表示一个长度为 n 的数组
// 代码中的 array.length 就等于 n
int[] array = new int[n];
int count = 0;

void insert(int val) {
if (count == array.length) {
int sum = 0;
for (int i = 0; i < array.length; ++i) {
sum = sum + array[i];
}
array[0] = sum;
count = 1;
}

array\[count\] = val;
++count;

}

这段代码的意思是，如果数组没有满，就直接将数据插入数组；如果满了，就先将数组遍历求和，将和存入数组第一个元素，再插入后面的元素。很明显，最好和最坏时间复杂度分别为O(1)和O(n)，而平均复杂度为O(1)。像这种大部分情况复杂度很低，极端情况下复杂度很高的代码，我们可以将复杂度高的部分分摊到低的部分。因此，一般均摊时间复杂度等于最好情况时间复杂度。 （反正也不会用，溜了溜了）
